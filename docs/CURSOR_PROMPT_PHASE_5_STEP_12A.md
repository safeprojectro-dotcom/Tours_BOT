Фаза: Phase 5 / Step 12A — Telegram chat cleanup / message hygiene

Почему идём сюда
- Mini App happy-path уже работает
- bookings/payment UX уже укреплён
- но в Telegram private chat накапливаются старые служебные сообщения, кнопки, language picker, списки туров и ответы на команды
- это начинает путать пользователя и ухудшает восприятие продукта

Проблема
Сейчас бот слишком часто добавляет новые сообщения в ленту вместо переиспользования / обновления актуального служебного состояния.
Нужно уменьшить шум, не ломая flow и не удаляя важные пользовательские якоря.

Цель этого среза
Реализовать безопасную message hygiene policy:
1. transient / service messages можно удалять или заменять
2. важные итоговые сообщения не трогать
3. текущий home / CTA / browse state должен быть понятнее и компактнее

Жёсткие границы scope
Не менять:
- бизнес-логику бронирования
- payment flow
- webhook
- Mini App API
- schema / migrations
- admin/operator flows
- waitlist

Разрешённый scope
- app/bot/handlers/private_entry.py
- app/bot/keyboards.py
- app/bot/messages.py
- app/bot/services.py
- при необходимости новый небольшой helper/state-файл для хранения message ids
- docs
- узкие unit tests при необходимости

Минимальный безопасный срез
1. Определить, какие сообщения считаются transient/service:
   - language picker
   - промежуточные prompt-сообщения для фильтрации
   - browse/list messages, которые устаревают
   - стартовые CTA-сообщения, если отправлены повторно
2. В private chat ввести безопасную политику:
   - при отправке нового служебного home/browse/prompt-сообщения попытаться удалить предыдущее аналогичное служебное сообщение для этого user/chat
   - если удалить нельзя — молча пропустить и продолжить
3. Не удалять:
   - пользовательские сообщения
   - подтверждение оплаты / важные итоговые статусы
   - support/contact meaningful replies
4. Для хранения достаточно лёгкого механизма:
   - in-memory map / FSM / per-user ephemeral pointer
   - без миграций и БД
5. Обновить docs и CHAT_HANDOFF

Ожидаемый результат
- в private chat остаётся меньше мусора
- повторный /start, /tours, /language, фильтры не создают длинную ленту однотипных служебных сообщений
- основные CTA к Mini App остаются понятными
- важные финальные сообщения не исчезают

Предпочтительный подход
- сначала определить 1–2 категории сообщений для cleanup, а не пытаться охватить всё
- safest slice: home/start + language prompt + transient filter prompts
- удаление делать best-effort через try/except

Проверки
- py_compile изменённых файлов
- ручной smoke:
  1. /start дважды -> не появляется мусорная дублирующая лента
  2. /language несколько раз -> старый picker не накапливается
  3. фильтр по дате/направлению/бюджету -> старые промежуточные prompts не остаются пачкой
  4. важные итоговые сообщения не исчезают
- полный suite не запускать
