## Concept

Create a read-only readiness object, for example:

publish_readiness:
  status: "ready_to_suggest" | "blocked" | "not_applicable" | "already_published" | "needs_review"
  recommended_action: string
  gates_passed_count: int
  gates_failed_count: int
  gates_warning_count: int
  gates:
    - code: string
      label: string
      status: "passed" | "failed" | "warning" | "not_applicable"
      reason: optional string
      severity: "blocker" | "warning" | "info"
  can_suggest_manual_publish: bool
  can_auto_publish: false
  auto_publish_mode: "disabled"
  publish_action_path: optional string
  prepare_conversion_chain_plan_path: optional string
  generated_at: datetime

Use existing schema/style conventions if the project already has similar read models.

## Status logic

Suggested top-level statuses:

- already_published:
  - offer lifecycle already published
  - can_suggest_manual_publish=false
  - reason: already published

- ready_to_suggest:
  - all required gates passed
  - not already published
  - can_suggest_manual_publish=true
  - can_auto_publish=false

- needs_review:
  - only warning/non-blocking gates remain
  - admin review recommended
  - can_suggest_manual_publish may be false unless current policy allows warnings

- blocked:
  - one or more blocker gates failed
  - can_suggest_manual_publish=false

- not_applicable:
  - row/type is not supplier-offer publish candidate

Adapt naming if existing enum style differs.

Important:
- Since many current offers may already be `published`, readiness should not tell admin to publish them again.
- For published offers, readiness may say “already published; use wider funnel / conversion readiness instead.”
