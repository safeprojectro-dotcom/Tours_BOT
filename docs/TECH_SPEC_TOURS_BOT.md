# Техническое задание

## Проект: Telegram-бот для продаж автобусных туров 1–2 дня

### Версия документа

v1.0

### Статус

Утверждено к реализации

---

# 1. Цель проекта

Разработать единый Telegram-бот для продажи автобусных туров 1–2 дня, который:

* работает в Telegram-группе как AI-ассистент для прогрева аудитории;
* переводит пользователей в личный диалог для продажи;
* открывает Mini App на Flet для оформления брони и оплаты;
* использует единую витрину туров из админ-панели;
* автоматизирует публикацию контента в Telegram и соцсети после утверждения администратором;
* минимизирует участие менеджера в рутинных продажах;
* передает сложные кейсы человеку через handoff;
* является мультиязычным и готовым к масштабированию.

Проект должен быть реализован с учетом дальнейшего расширения до production-архитектуры с RAG, расширенной аналитикой, waitlist, повторными продажами и автоматизацией контента.

# 2. Бизнес-модель и логика продаж

## 2.1 Общая модель

Используется один основной клиентский бот.

Он же:

* отвечает в Telegram-группе;
* общается с пользователем в личке;
* открывает Mini App;
* ведет к оплате;
* отправляет уведомления и напоминания.

## 2.2 Роли каналов

Назначение:

* прогрев;
* ответы на типовые вопросы;
* вовлечение;
* социальное доказательство;
* подводка к продаже.

### Личный чат с ботом

Назначение:

* подбор тура;
* уточнение параметров;
* бронь;
* перевод в Mini App;
* оплата;
* подтверждение;
* постпродажные уведомления.

### Mini App

Назначение:

* полноценная витрина туров;
* просмотр карточки тура;
* выбор количества мест;
* выбор точки посадки;
* оформление заказа;
* запуск оплаты;
* просмотр статуса заказа.

### Менеджер / оператор

Подключается только в сложных случаях.

# 4. Технологический стек

## 4.1 Обязательный стек

* Python
* aiogram — Telegram Bot API
* Flet — Mini App / интерфейс каталога и бронирования
* FastAPI — backend API
* PostgreSQL — основная БД
* Redis — кеш, очереди, rate limiting, временные блокировки, дедлайны брони
* SQLAlchemy / SQLModel — ORM
* Alembic — миграции БД
* APScheduler / Celery / RQ — фоновая обработка уведомлений, дедлайнов, напоминаний
* pgvector или отдельная vector DB — на этапе production

## 4.2 Архитектурный стиль

Рекомендуемая структура: модульный backend с четким разделением:

* bot layer;
* mini app layer;
* domain/services;
* repository/data layer;
* content generation layer;
* analytics layer;
* handoff/operator layer.

# 7. Пользовательские сценарии

## 7.1 Сценарий: интерес в группе

1. Пользователь пишет в группе вопрос о туре.
2. Бот определяет интент.
3. Бот отвечает кратко и по существу.
4. Бот предлагает перейти в личку или открыть Mini App.
5. Пользователь переходит в private flow.

## 7.2 Сценарий: продажа в личке

1. Пользователь пишет боту в личку.
2. Бот определяет язык и/или предлагает выбрать язык.
3. Бот уточняет ключевые параметры:

   * дата;
   * направление;
   * количество мест;
   * город/точка посадки.
4. Бот предлагает 1–3 тура.
5. Пользователь выбирает тур.
6. Бот временно удерживает места.
7. Бот отправляет пользователя в Mini App или подтверждает бронь в чате.
8. Пользователь оплачивает.
9. Бот подтверждает заказ.
10. Бот отправляет дальнейшие уведомления.

## 7.3 Сценарий: бронирование через Mini App

1. Пользователь открывает Mini App.
2. Видит каталог туров.
3. Открывает карточку тура.
4. Указывает количество мест и точку посадки.
5. Подтверждает бронь.
6. Получает таймер бронь-до.
7. Запускает оплату.
8. После оплаты получает подтверждение.

## 7.4 Сценарий: нет мест

1. Пользователь пытается открыть тур без мест.
2. Система предлагает:

   * waitlist;
   * альтернативный тур;
   * другую дату.
3. Пользователь может подписаться на освобождение места.

## 7.5 Сценарий: сложный кейс

1. Бот понимает, что нужен человек.
2. Создает handoff.
3. Оператор получает уведомление.
4. Оператор продолжает диалог.
5. После завершения кейса может вернуть диалог боту.

# 8. Логика поведения бота в группе

## 8.1 Общие правила

Бот в группе:

* отвечает на команды;
* отвечает на упоминания;
* реагирует на ключевые продающие триггеры;
* не спамит;
* не публикует длинные сообщения без необходимости;
* не просит личные данные публично;
* не ведет длинные персональные переговоры в группе.

## 8.2 Примеры триггеров

* цена / сколько стоит;
* есть места;
* когда выезд;
* откуда забираете;
* хочу поехать;
* 2 места;
* можно забронировать;
* условия оплаты;
* есть тур на выходные.

## 8.3 Ответ в группе должен

* быть кратким;
* содержать конкретику;
* завершаться CTA;
* переводить к личному контакту или Mini App.

## 8.4 Настройка активности

В MVP активность фиксированная.
Во второй фазе предусмотреть переключаемые режимы:

* низкая активность;
* стандартная активность;
* высокая активность.

# 9. Логика продаж в личке

## 9.1 Принципы

* короткие сообщения;
* один шаг за раз;
* не задавать много вопросов одновременно;
* предлагать 1–3 варианта;
* использовать понятный CTA;
* не выдумывать данные;
* при низкой уверенности переводить в handoff.

## 9.2 Обязательные сценарии

* подбор тура по дате;
* подбор тура по направлению;
* подбор тура по бюджету;
* бронирование;
* контроль дедлайна;
* запуск оплаты;
* проверка оплаты;
* напоминания;
* подтверждение;
* post-trip сообщение;
* предложение следующего тура.

# 10. Mini App: функциональные требования

## 10.1 Общие требования

* галерею;

* остаток мест;

* выбрать точку посадки;
* видеть дедлайн удержания брони;

## 10.7 Экран: мои брони

Должен показывать:

* активные брони;
* подтвержденные заказы;
* canceled;
* waitlist;
* статус оплаты;
* кнопки связи/помощи.

# 12. Витрина туров и контентный конвейер

## 12.1 Единая точка правды

Вся информация о турах хранится в одной системе — админке/БД.

Нельзя хранить цены, даты, статусы и ключевые данные разрозненно.

## 12.2 Источник контента

Контент для:

* Telegram-группы;
* Telegram-бота;
* Mini App;
* Facebook;
* Instagram;
* TikTok

должен генерироваться на основе одной записи тура.

## 12.3 Контент-ассистент должен уметь генерировать

* карточку тура для бота;
* текст для Telegram-группы;
* пост для Facebook;
* подпись/текст для Instagram;
* сценарий/структуру для TikTok/Reels;
* короткие CTA;
* вариант для «последние места»;
* вариант для waitlist;
* мультиязычные версии.

## 12.4 Публикация

Логика в MVP:

1. Админ создает тур.
2. Ассистент генерирует набор контента.
3. Админ утверждает.
4. После утверждения ассистент публикует в заданные каналы.

# 13. Мультиязычность

## 13.1 Общие требования

Архитектура должна быть multilingual-ready.

## 13.2 Минимальная логика

* бот определяет язык пользователя по Telegram locale, данным профиля или первому сообщению;
* при необходимости предлагает выбор языка;
* сохраняет язык пользователя в профиле;
* все шаблоны должны иметь языкозависимую структуру.

## 13.3 Языки

Архитектура должна поддерживать расширяемый набор языков.
Стартовые языки не должны быть захардкожены как окончательный список.

Система должна быть готова как минимум к таким языкам:

* румынский;
* русский;
* сербский;
* венгерский;
* итальянский;
* немецкий;
* английский.

## 13.4 Контент

Туры могут иметь:

* master-запись;
* мультиязычные названия;
* мультиязычные описания;
* мультиязычные шаблоны публикаций.

# 15. Политика брони и оплаты

## 15.1 Основная модель

* пользователь выбирает тур;
* система удерживает место/места ограниченное время;
* в течение окна брони пользователь должен произвести полную оплату;
* если оплаты нет, бронь автоматически снимается.

## 15.2 Таймер брони

Для MVP использовать следующую логику:

* выезд через 1–3 дня: бронь 2–6 часов;
* выезд через 4–14 дней: бронь до 24 часов;
* для горячих дат должен поддерживаться настраиваемый жесткий дедлайн.

## 15.3 После оплаты

После успешной оплаты заказ получает статус confirmed.

## 15.4 При просрочке оплаты

* бронь снимается;
* статус заказа обновляется;
* места возвращаются в пул;
* пользователю отправляется уведомление;
* при необходимости предлагается повторная бронь или другой тур.

# 16. Waitlist

## 16.1 Обязательность

Waitlist обязателен уже в MVP.

## 16.2 Возможности

Если мест нет, бот должен:

* предложить вступить в waitlist;
* предложить альтернативный тур;
* предложить другую дату.

## 16.3 Уведомления

Если место освобождается:

* пользователю из waitlist должно прийти уведомление;
* место должно резервироваться по логике очереди или по подтвержденному бизнес-правилу.

# 17. Handoff на человека

## 17.1 Handoff обязателен если:

* группа больше 4–6 человек;
* пользователь просит скидку;
* нестандартная точка посадки;
* индивидуальный маршрут;
* конфликт/жалоба;
* сложный перенос/отмена;
* проблема с оплатой;
* бот не уверен в ответе;
* пользователь просит человека напрямую.

## 17.2 Требования к handoff

Система должна:

* создавать запись handoff;
* фиксировать причину;
* назначать приоритет;
* сохранять историю контекста;
* уведомлять оператора;
* помечать чат как находящийся у оператора;
* позволять вернуть чат боту.

# 18. Платежный модуль

## 18.1 Общие требования

В ТЗ закладывается абстрактный платежный модуль.

Система не должна быть архитектурно привязана к одному провайдеру.

## 18.2 Интерфейс платежного модуля

Должен поддерживать:

* создание счета / ссылки / платежного намерения;
* webhook подтверждения;
* проверку статуса;
* отмену платежа;
* возврат.

## 18.3 Технические требования

* модуль должен быть заменяемым;
* провайдер должен подключаться через adapter pattern;
* платежи должны логироваться;
* ошибки платежей должны обрабатываться безопасно.

* пользователь переходит в личку и/или Mini App;
* система удерживает бронь;
* пользователь оплачивает;
* заказ подтверждается;
* бот сопровождает клиента до поездки;
* сложные случаи передаются оператору;
* контент по туру автоматически готовится и публикуется после утверждения;
* вся аналитика и статусы находятся в одной системе.

# 32. Формулировка задачи для Cursor

Нужно реализовать production-ready архитектуру MVP Telegram-бота для продаж автобусных туров 1–2 дня на Python с использованием aiogram, FastAPI, PostgreSQL и Flet Mini App.

Обязательные характеристики:

* один клиентский бот для группы и лички;
* единая витрина туров в админке;
* продажа через личку + Mini App;
* ограниченная по времени бронь;
* полная оплата;
* waitlist;
* handoff;
* мультиязычность;
* контент-ассистент;
* автопубликация после утверждения;
* модульная расширяемая архитектура.

Реализацию вести поэтапно, без ломки архитектуры, с акцентом на чистый код, расширяемость и дальнейшее подключение RAG.
